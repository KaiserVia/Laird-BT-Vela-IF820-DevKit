# LPC1766 – Projektspezifische Dokumentation

---

## UART

### Übersicht der drei UARTs

Der LPC1766 stellt drei UART-Peripherien bereit. Im Projekt werden alle drei genutzt, aber unterschiedlich:

| Peripheral | Pins (PINSEL) | Zweck im Projekt | Baudrate | ISR |
|---|---|---|---|---|
| UART0 | PINSEL0: TXD0, RXD0 | RS232-Terminal (immer aktiv) | 115200, fest | `UART0_IRQHandler` |
| UART1 | PINSEL4: TXD1, RXD1, CTS1, RTS1 | BT / GSM / GPS via Hardware-MUX | variabel (9600 / 115200 / 307200 / 460800) | `UART1_IRQHandler` |
| UART2 | PINSEL0: TXD2, RXD2 | GPS (direkte Verbindung, kein MUX) | 9600 / 115200 / 460800 | `UART2_IRQHandler` |

---

### UART0 – RS232-Terminal

UART0 ist fest auf 115200 Baud konfiguriert und wird nie deinitialisiert.

Konfiguration in `hard.c: Init_UART0()`:

```c
PCONP |= PCUART0;           // Power ein
U0->IER = 0x00;             // Interrupts disable
U0->FCR = 0x07;             // FIFO enable + reset
U0->LCR = 0x00000083;       // 8N1, DLAB=1
U0->DLL = 0x04;             // Ganzzahlteiler fuer 115200 Baud bei 24 MHz
U0->LCR = 0x00000003;       // 8N1, DLAB=0
U0->FDR = (8<<4)|0x05;      // Fractional divider 5/8
U0->IER = 0x01;             // RX-Interrupt enable
NVIC_SetPriority(UART0_IRQn, 30);
NVIC_EnableIRQ(UART0_IRQn);
```

NVIC-Prioritaet: 30 (niedrigste der drei UARTs).

---

### UART1 – Multiplexed: BT / GSM / GPS

UART1 ist die zentrale Kommunikationsschnittstelle fuer alle externen Module. Physisch wird sie per Hardware-MUX (IC55/IC56) auf das jeweils aktive Geraet geschaltet. Die Baudrate muss bei jedem MUX-Wechsel neu gesetzt werden.

#### Hardware-MUX

Die Kanalauswahl erfolgt ueber zwei GPIO-Pins auf Port 0 (`CSAF` = P0.1, `CSBF` = P0.0):

```c
// hard.h
#define MUX_BT_Select   FIO0CLR=CSAF|CSBF          // BT-Kanal: beide Pins LOW
#define MUX_GSM_Select  FIO0CLR=CSBF; FIO0SET=CSAF // GSM-Kanal: CSAF HIGH, CSBF LOW
#define MUX_NO_Select   FIO0SET=CSAF|CSBF          // kein Kanal: beide Pins HIGH
```

**Wichtig:** Diese Makros schalten NUR den GPIO-Pin. Die UART1-Baudrate bleibt unveraendert. Wer nur den MUX umschaltet ohne `Init_UART1()` zu rufen, sendet/empfaengt danach mit der alten Baudrate – das fuehrt zu Zeichensalat beim naechsten Geraet.

#### Bautdraten je Geraet

| Geraet | Betriebsbaudrate |
|---|---|
| Laird 730-SA (BT) | 460800 |
| Microchip RN4678APL (BT) | 307200 |
| Quectel EG91 (LTE) | 460800 |
| Quectel UC15 (UMTS) | 460800 |
| Alle Geraete Erstinitialisierung | 115200 oder 9600 |

#### Init_UART1() – Baudratenkonfiguration

`Init_UART1(baud)` in `hard.c` setzt die UART1-Register fuer die gewuenschte Baudrate (Takt: 24 MHz, PCLK = CCLK):

```c
// baud == 0: Deinitialisierung
NVIC_DisableIRQ(UART1_IRQn);
PINSEL4 &= ~(TXD1|RXD1|CTS1|RTS1);  // Pins wieder als GPIO
PCONP   &= ~PCUART1;                  // Power aus
connect &= ~UART1;
MUX_NO_Select;

// baud == 9600 (+0,15%)
U1->DLL = 0x68;  U1->FDR = (2<<4)|0x01;   // Fractional 1/2

// baud == 115200 (+0,16%)
U1->DLL = 0x08;  U1->FDR = (8<<4)|0x05;   // Fractional 5/8

// baud == 307200 (+0,16%)
U1->DLL = 0x03;  U1->FDR = (8<<4)|0x05;   // 24MHz/(16*3*1.625)

// baud == 460800 (+1,6%)
U1->DLL = 0x03;  U1->FDR = (12<<4)|0x01;  // Fractional 1/12
```

Format immer 8N1. Hardware-Flowcontrol RTS/CTS ist aktiv:
```c
U1->MCR |= (1<<7)|(1<<6);  // Enable RTS/CTS flow control
```

NVIC-Prioritaet: 29.

#### CTS-Check nach Initialisierung

`Init_BT_ch()` prueft nach dem Init, ob das Modul sendebereit ist (CTS = LOW = bereit):

```c
if ((FIO2PIN & CTS) == 0) return TRUE;  // Modul bereit
return FALSE;                            // Modul nicht bereit
```

Bei `rbuf_full` (Pufferueberlauf) wird der UART1-Interrupt deaktiviert. RTS geht dann automatisch HIGH und stoppt das sendende Geraet (Hardware-Flowcontrol). Der Interrupt wird erst wieder aktiviert, wenn der Puffer geleert wurde.

#### Richtiges Muster fuer MUX-Wechsel

Immer beide Schritte zusammen ausfuehren – nie nur den MUX allein:

```c
// Richtig:
Init_BT_ch(BT_BAUD);    // = MUX_BT_Select + Init_UART1(baud) + CTS-Check
Init_GSM_ch(460800);    // = MUX_GSM_Select + Init_UART1(460800)

// Falsch (Baudrate bleibt auf altem Wert):
MUX_BT_Select;          // nur MUX-Pin, keine Baudrate!
```

### UART2 – GPS (Direktverbindung)

UART2 verbindet direkt mit dem GPS-Modul (L70 / LC76F) ohne Hardware-MUX.

Konfiguration analog UART0/1, unterstuetzte Baudraten: 9600, 115200, 460800.  
Deinitialisierung mit `Init_UART2(0)`.

NVIC-Prioritaet: 31 (niedrigste im System).

---

### Empfangspuffer und Indizes

Der gesamte UART0/1-Empfang laeuft ueber einen gemeinsamen 256-Byte-Ringpuffer (`rbuf`). UART2 hat einen eigenen GPS-Puffer (`gbuf`). MQTT-Daten werden in `pbuf` geschrieben.

| Puffer | Groesse | Genutzt von | Deklaration |
|---|---|---|---|
| `rbuf[PAGE]` | 256 Byte | UART0 und UART1 | `hard.c:40` |
| `pbuf[PAGE]` | 256 Byte | UART1 MQTT-Modus und Plus-Matrix | `hard.c:41` |
| `gbuf[PAGE]` | 256 Byte | UART2 (GPS) | `hard.c:88` |
| `mqbf[MQBFSIZE]` | 3328 Byte | MQTT Empfang | `hard.c` |

#### Indizes

```c
volatile uchar rxi;   // Schreibindex ISR -> rbuf[++rxi] = U0/U1->RBR
volatile uchar bxi;   // Leseindex Hauptprogramm -> rbuf[++bxi]
volatile uchar rx2;   // Schreibindex ISR UART2 -> gbuf[++rx2]
volatile uchar bx2;   // Leseindex Hauptprogramm UART2
volatile uchar rpi;   // Schreibindex MQTT/Plus-Puffer pbuf
volatile uchar bpi;   // Leseindex MQTT/Plus-Puffer pbuf
```

Alle Indizes sind `uint8` (0..255), der Ueberlauf von 255 auf 0 ist gewollt und bildet den Ring.

#### Puffer-Flush

```c
bxi = rxi;   // Verwirft alle bisher empfangenen, noch ungelesenen Zeichen in rbuf
bx2 = rx2;   // dto. fuer UART2/GPS-Puffer
```

Das ist die einzige korrekte Methode, den Empfangspuffer zu leeren. **Niemals** `rxi` zuruecksetzen – das wuerde in den laufenden ISR-Schreibpfad eingreifen.

---

### ISR-Logik

Alle drei ISRs folgen demselben Muster: Auswertung von `IIR` (Interrupt Identification Register), Zeichen aus `RBR` lesen, in den jeweiligen Ringpuffer schreiben.

**UART1-ISR besonderheiten:**

- Wenn eine MQTT-Verbindung aktiv ist (`connect & MQTT_LINK`), wird das empfangene Zeichen in `pbuf` geschrieben, nicht in `rbuf`.
- Bei Pufferueberlauf wird der UART1-Interrupt komplett abgeschaltet und `rbuf_full = 1` gesetzt. Hardware-RTS stoppt das sendende Geraet.
- Jedes empfangene Zeichen setzt `bt_time = BT_TIMEOUT` (6 Minuten) – damit wird der Bluetooth-Verbindungstimer zurueckgesetzt.

```c
// UART1_IRQHandler (vereinfacht)
case 0x0004:
    if (connect & MQTT_LINK)
        pbuf[++rpi] = U1->RBR;          // MQTT-Pfad
    else if (bxi != (rxi+1))            // Platz im Puffer?
    {
        rbuf[++rxi] = U1->RBR;
        bt_time = BT_TIMEOUT;           // Verbindungstimer reset
    }
    else
    {
        rbuf_full = 1;
        NVIC_DisableIRQ(UART1_IRQn);    // Flusssteuerung: UART1 stopp
    }
```

---

### Sende-Pfad (Polling)

Das Senden erfolgt nicht per Interrupt, sondern per Polling auf das `THRE`-Flag (Transmitter Holding Register Empty). Das ist in `sio.c: putb()` implementiert:

```c
// Senden auf UART1:
while (!(U1->LSR & THRE)) { /* warte */ }
U1->THR = ch;
```

Bei UART1 gibt es einen Watchdog-Schutz: Wenn `THRE` nicht innerhalb von 1 ms vor dem Watchdog-Reset frei wird, bricht `putb()` mit `-1` ab.

---

### connect-Flag

Das globale `connect`-Byte steuert, an welche Schnittstellen `putb()` ausgibt:

```c
#define UART0    (1<<0)   // RS232
#define UART1    (1<<1)   // BT / GSM (aktiver MUX-Kanal)
#define UART2    (1<<2)   // GPS
#define USB_LINK (1<<3)
#define MQTT_LINK (1<<5)
```

Vor einem Geraete-Dialog wird `connect` auf `UART1` gesetzt, danach auf den gespeicherten Wert `concpy` zurueckgestellt.

---

### Deinit – wann und warum

`Init_UART1(0)` wird immer aufgerufen, wenn keine Kommunikation mit BT/GSM stattfindet. Das ist wichtig, weil:

- PINSEL4-Pins werden auf GPIO (Input, Pull-up) zurueckgestellt – kein floating Input an der UART.
- `PCONP &= ~PCUART1` schaltet den UART-Takt ab – Stromersparnis.
- `MUX_NO_Select` legt beide MUX-Pins auf HIGH – kein Geraet ausgewaehlt.
- Der UART1-IRQ ist abgeschaltet – keine Phantom-Interrupts durch Leitungsrauschen.

## Laird BT730-SA – Modulreferenz

Quelle: `Doku/BT730 (Hersteller Laird)/_BT730 Befehle (Zusammenfassung).txt`

### Baudrate und Werkseinstellung

- **Default-Baudrate nach Power-on oder Werksreset: 9600 Baud**
- `AT&F1` (im Projekt verwendet): Werksreset, Baudrate bleibt auf 9600
- `AT&F3` (Alternative, nicht im Projekt): Werksreset, Baudrate wechselt auf 115200
- Das erklaert, warum der Initialisierungscode in `btio.c` zuerst mit 460800 versucht, bei Ausbleiben der Antwort auf 9600 zurueckfaellt und dann mit `ATS521=460800` auf die Betriebsbaudrate hochstellt.

### Stromverbrauch

| Betriebszustand | Strom (bei VCC = 3,8 V) |
|---|---|
| Sniff Mode (S561-S565 konfiguriert) | 1,8 mA |
| Idle Mode (S512=1) | 1,25 mA |
| Connected Mode, max. Datentransfer | 35 mA |
| Connecting Mode (ATDxxx) | 66 mA |

### S-Register die im Projekt gesetzt werden

| Register | Wert | Bedeutung |
|---|---|---|
| S502 | 1 | Authentifizierung fuer eingehende Verbindungen aktivieren |
| S504 | 1 | Autoresponse; CONNECT/NO CARRIER/RING-Meldungen unterdruecken (Cable-Replacement-Modus) |
| S507 | 2 | `^^^`-Escape-Sequenz deaktivieren (verhindert Durchsatz-Einbruch, siehe unten) |
| S508 | 1000 | Page Scan Interval 1 Sekunde |
| S512 | 4 | Power-up state: connectable und discoverable (entspricht AT+BTP) |
| S514 | 60 | Pairing-Timeout in Sekunden |
| S521 | 460800 | Baudrate auf 460800 setzen (sofort wirksam, wird bei AT&W gespeichert) |
| S538 | 1 | Erfolgreiches Pairing automatisch in Trusted-Device-Datenbank speichern |

### Warum S507=2 (^^^ deaktivieren)

Das Aktivieren der `^^^`-Escape-Sequenz halbiert den Datendurchsatz:

> "Being able to drop a connection using the escape sequence ^^^ has a severe penalty on data throughput. In fact, the data rate is of the order of 85 kbps instead of about 200 kbps."

Durch `ATS507=2` wird die ^^^-Erkennung abgeschaltet. Verbindungstrennung laeuft stattdessen ueber DTR/CTS oder Stromabschaltung des Moduls.

### Debugging-Befehle (Abfragen)

```

```

## LTE-Modem (Quectel UC15 / EG91) – Baudraten und Initialisierung

### AT+IPR – Baudrate einstellen (Quelle: EG9x AT Commands Manual, §3.5)

Command `AT+IPR` sets the baud rate of the UART. The default baud rate is **115200 bps**.

| Befehl | Beschreibung |
|--------|-------------|
| `AT+IPR=?` | Zeigt unterstuetzte Baudraten |
| `AT+IPR?` | Liest aktuelle Baudrate |
| `AT+IPR=<rate>` | Setzt Baudrate (wirksam nach OK) |
| `AT+IPR=<rate>;&W` | Setzt und speichert in einem Schritt ← **im Projekt verwendet** |

Unterstuetzte Baudraten (EG91):

| Baudrate | Unterstuetzt |
|----------|-------------|
| 9600 | ja |
| 19200 | ja |
| 38400 | ja |
| 57600 | ja |
| 115200 | ja ← **Default** |
| 230400 | ja |
| **460800** | **ja ← Im Projekt verwendet** |
| 921600 | ja |
| 2900000 | ja |
| 3000000 | ja |

**Wichtige Hinweise:**
- `AT+IPR` wird **nicht** durch `AT&F` oder `ATZ` zurueckgesetzt.
- Speichern nur mit `AT&W` (oder kombiniert `AT+IPR=<rate>;&W`).
- Im Multiplex-Modus kann `AT+IPR` nicht geaendert werden.
- Die neue Baudrate ist sofort nach dem `OK` aktiv.

### Betriebsbaudrate im Projekt

- **Festgelegte Betriebsbaudrate: 460800 Baud**
- Gespeichert im Modem mit: `AT+IPR=460800;&W` (= `T_cfsto` in `gsmio.h`)

### Baudrate-Stellen im Code

| Datei | Stelle | Beschreibung |
|-------|--------|-------------|
| `gsmio.h` | `#define T_cfsto` | AT-Kommando das Baudrate ins Modem speichert |
| `gsmio.c: gsm_power()` | `Init_GSM_ch(460800)` | UART1-Baudrate beim Einschalten/MUX-Wechsel |
| `gsmio.c: init_gsm()` | `Init_GSM_ch(460800)` | UART1-Baudrate bei Erst-Initialisierung |

```

  OK? -> weiter
  kein OK bei i==5: Init_UART1(115200)   -> Fallback 115200
  kein OK bei i==3: Init_UART1(460800)   -> Fallback 460800 (Altbestand)
  kein OK bei i==1: Init_UART1(9600)     -> Fallback 9600

put2str(AT,&F)             -> Werksreset AT&F (bis zu 3x wiederholen)

```

### Betrieb (nach Initialisierung, `gsm_power(2)` oder `gsm_power(3)`)

```

FIO0CLR = GSM_DTR          -> Sleep disable
Warte auf CTS LOW          -> Modem empfangsbereit (max. 10s, 200 x 50ms)
  kein CTS: return(-1)     -> Fehler

// Wenn kein Neustart: AT+CFUN? abfragen (bis zu 16x)
putln(AT+CFUN?)            -> Funktionsstatus abfragen
  +CFUN: 1 empfangen -> return(1) -> Kommunikation moeglich

// Nach Neustart: Auf +CFUN: 1 URC warten (max. 15s)
  +CFUN: 1 empfangen -> return(1)
  Timeout -> mqtt_con_err=44, return(-1)

// Nach erfolgreichem gsm_power(2/3): MUX bleibt auf GSM.
// Am Ende der Kommunikation immer:
Init_UART1(0)              -> Deinitialisieren
MUX_BT_Select              -> MUX zurueck auf BT (Default-Zustand)
```

- **LTE (UC15/EG91): Nur 460800 Baud funktioniert zuverlässig.** Andere Baudraten (z.B. 307200) fuehren zu Kommunikationsfehlern.

## Recovery-Massnahmen bei Baudrate-Mismatch

### Allgemeines Prinzip

Wenn ein Modul nicht antwortet, startet ein Fallback-Loop, der alle bekannten Baudraten durchprobiert:

**Reihenfolge: 307200 → 115200 → 460800 → 9600**

Das deckt ab:
- Normalbetrieb (307200 / 460800 je Modul)
- Frisch geflashtes / zurückgesetztes Modul (115200 oder 9600)
- Altbestand mit alter Firmware (460800)

### Laird 730-SA Recovery (`init_bluetooth`, `retry=4`)

```
Init_BT_ch(460800)           -> Erster Versuch auf Betriebsbaudrate
bt_command(AT&F1)            -> OK?
  Fehler, retry==3: Init_UART1(115200)  -> 2. Versuch
  Fehler, retry==2: Init_UART1(460800)  -> 3. Versuch (deckt Altbestand ab)
  Fehler, retry==1: Init_UART1(9600)    -> 4. Versuch (Werksauslieferung)

// Wenn Antwort erhalten (bei beliebiger Baudrate):
Init_UART1(9600)             -> Zurueck auf 9600 fuer sichere Konfiguration
bt_command(ATE0)             -> Echo aus
putstr(ATS521=460800)        -> Neue Baudrate bekanntgeben (noch auf 9600)
Init_UART1(460800)           -> Host wechselt auf Zielbaudrate
wait_ok_time(300)            -> OK empfangen? Dann Modul ist jetzt auf 460800
```

### RN4678 Recovery (`init_bluetooth`, `retry=4`)

```
Init_BT_ch(307200)           -> Erster Versuch auf Betriebsbaudrate
bt_command($$$)              -> CMD> empfangen?
  Fehler, retry==3: Init_UART1(115200)  -> Default nach SF,1
  Fehler, retry==2: Init_UART1(460800)  -> Fallback
  Fehler, retry==1: Init_UART1(9600)    -> Letzter Versuch

// Wenn CMD> empfangen (bei beliebiger Baudrate):
bt_command(SF,1)             -> Factory Reset
bt_command(R,1)              -> Reboot
Init_BT_ch(115200)           -> Modul startet nach SF,1 immer auf 115200
// ... vollstaendige Neukonfiguration inkl. SU,10 + Reboot auf 307200
```

### LTE Recovery (`init_gsm`, `i=7`)

```
Init_GSM_ch(307200)          -> Erster Versuch auf Betriebsbaudrate
putln(AT)                    -> OK?
  i==5: Init_UART1(115200)   -> 2. Versuch
  i==3: Init_UART1(460800)   -> 3. Versuch
  i==1: Init_UART1(9600)     -> 4. Versuch

// Wenn OK empfangen (bei beliebiger Baudrate):
put2str(AT,&F)               -> Werksreset (stellt sicher, dass Konfiguration sauber)
// ... AT+IPR=460800;&W speichert Zielbaudrate fuer naechsten Start
```

### MUX-Wechsel im Betrieb (kein Neustart, `Init_BT_ch` / `Init_GSM_ch`)

Jedes Mal, wenn UART1 von einem Geraet auf ein anderes umgeschaltet wird:

```c
// Korrekte Sequenz (in Init_BT_ch / Init_GSM_ch):
osDelay(3);              // Warte bis letztes gesendetes Byte raus ist
MUX_xx_Select;           // GPIO-Pins umschalten
Init_UART1(ziel_baud);   // UART1 auf neue Baudrate konfigurieren
osDelay(3);              // Warte auf Leitungsstabilisierung

// Danach Puffer leeren (verhindert Artefakte vom vorherigen Geraet):
bxi = rxi;
```
